# 2026-04-06 Navly 数据中台重构方案

日期：2026-04-06  
状态：draft-for-review  
目标读者：项目 owner、数据研发、后续重构执行者、外部审核 AI

---

## 1. 文档目的

本文档定义 **Navly** 项目的数据中台重构方案。

本次方案不是对现有“业务问答系统”做局部修补，而是把项目中的数据相关能力抽离、重组、重命名、重新分层，建设一个可长期复用的**数据中台内核**。

重构完成后，项目中真正长期保留的资产将至少包括：

1. 数据中台内核
2. 基于已纳入仓库的 OpenClaw 源码与 WeCom 的权限、会话绑定与治理能力

其余现有上层业务编排、问答路由、技能胶水、深度分析链路，不再视为长期架构基石，可在后续阶段按新中台能力重新设计与实现。

补充背景：当前仓库已纳入上游 OpenClaw 源码目录 `upstreams/openclaw/`。后续权限内核与会话能力的保留、裁剪与复用，都应以仓内源码为基础进行，而不是再把 OpenClaw 视为外部黑盒依赖。

---

## 2. 最终目标

我们的最终目标不是“把 API 接进来”或“把表补满”，而是：

> 建立一个覆盖全部目标门店、支持全历史回溯、字段全量纳管、状态可解释、结果可投影、可被未来上层稳定消费的**数据中台**。

这个目标展开后包含六件事：

### 2.1 数据全量

- 覆盖全部白名单 / 目标门店
- 覆盖全历史业务日
- 覆盖 8 个正式 Qinqin API 接口
- 覆盖 API 文档中的全部字段

### 2.2 字段全量纳管

- API 文档字段不能被忽略
- 不是所有字段都必须立刻结构化，但所有字段都必须进入中台字段治理范围
- 每个字段都必须知道当前落点：原始层 / 事实层 / 主题层 / 暂不使用

### 2.3 同步状态可信

数据库必须能清楚回答：

- 某店某天哪些 endpoint 已跑
- 成功 / partial / failed / source_empty 的真实原因是什么
- 最新可用业务日是什么
- 是否存在历史 run 与当前状态表不一致

### 2.4 可答状态可信

数据库必须能清楚回答：

- 某个能力为什么 ready
- 为什么 pending
- 为什么 failed
- 是缺数据、缺权限、缺依赖、缺实现，还是补数未完成

### 2.5 主题消费稳定

未来上层系统默认不直接拼装底层事实，而是优先读取数据中台产出的主题结果 / projection / 服务对象。

### 2.6 可重构

未来上层重建时：

- 不需要重新发明数据口径
- 不需要重新发明同步状态
- 不需要重新发明可答判断
- 只需要围绕数据中台的稳定输出做新的业务编排

## 3. 设计原则

### 3.1 事实保真，主题供用，状态解释

数据中台必须同时满足三件事：

- **事实保真**：底层事实数据是唯一长期真相源
- **主题供用**：默认消费入口是主题层，而不是事实表散读
- **状态解释**：任意异常或缺口都必须可解释

### 3.2 API 文档字段全纳管

原则：

- API doc 字段全量纳管
- 已返回字段全量留存
- 是否结构化由读取策略决定
- 不能因为暂时不用就默默丢弃

当前 Qinqin API 输入文档主目录固定为：

- `docs/api/qinqin/`

该目录中的文档（共享认证、manifest、member/staff 分页文档）应被视为数据中台输入真相源之一，而不是可有可无的说明材料。  
与 live 行为差异相关的 gap audit / 恢复材料统一位于：

- `docs/audits/qinqin/`

### 3.3 一张表只表达一种真相

不能再让同一张表同时表达：

- 历史执行过程
- 当前状态
- 可答状态
- 主题结果

必须拆开。

### 3.4 新中台先于新上层

后续新业务上层必须建立在数据中台之上，而不是反过来继续倒逼数据层打补丁。

### 3.5 旧代码可复用，但必须服从新分层

现有代码如果经过改造后适合新中台，可以复用。

但复用条件是：

- 角色清晰
- 归属清晰
- 输入输出清晰
- 不把旧系统的混乱语义原样带入新目录

### 3.6 禁止把业务真相硬编码进实现

数据中台的长期目标是“可审计、可回放、可治理、可迁移”。

因此以下内容不应散落硬编码在脚本、orchestrator、prompt glue 或临时查询逻辑中：

- org / store / tenant / role / conversation 等标识
- endpoint 清单、字段白名单、字段映射口径
- sync 策略、时间窗、状态映射、回溯策略
- 权限判定、路由决策、主题选择逻辑
- 环境相关 URL、路径、header 组合或私有 target

推荐承载位置应为：

- contracts
- schema / field metadata
- registry / manifest
- 受治理的配置模块
- source-of-truth 状态表与目录表

如果阶段性必须保留兼容映射：

- 只能集中在单点模块
- 必须清楚标记为 legacy / temporary
- 必须配套文档说明其存在原因和替代方案

## 4. 方案对比

### 方案 A：继续在旧目录中渐进修补

做法：

- 保持现有目录结构
- 修 sync state
- 修 completeness
- 修 projection
- 修字段映射

优点：

- 起步快
- 改动表面较少

缺点：

- 数据中台与旧业务层继续纠缠
- 很难建立新的清晰边界
- 上层 bug 与数据层 bug 仍会互相污染
- 无法自然形成未来上层重构的稳定依赖面

结论：**不推荐**。

---

### 方案 B：建立新数据中台目录，旧代码择优迁移（推荐）

做法：

- 在新文件夹中建立数据中台
- 重新定义数据层分层与职责
- 现有代码按模块评估后迁移或重写
- 在中台稳定后，旧上层再逐步迁移到新中台输出

优点：

- 分层清晰
- 便于长期维护
- 便于外部审核
- 便于未来上层重建
- 便于字段治理和回溯治理

缺点：

- 前期要做目录与语义重估
- 会有一段新旧并行期

结论：**推荐方案**。

---

### 方案 C：全量推倒重写

做法：

- 旧数据代码全部废弃
- 从零做数据中台

优点：

- 架构最干净

缺点：

- 风险太高
- 复用价值被浪费
- 无法快速验证业务口径

结论：**不推荐作为当前阶段主策略**。

---

## 5. 推荐方案

采用 **方案 B：建立新数据中台目录，旧代码择优迁移**。

一句话概括：

> 不是在旧系统中“修数据层”，而是在新目录中“建设数据中台”；旧代码只是原材料，不是最终边界。

建议在仓库中新增独立目录，例如：

```text

    warehouse/
    quality/
    completeness/
```

也可命名为：

- `platforms/data-middle-platform`
- `platforms/data-kernel`
- `platforms/data-center`

推荐使用：

- `platforms/data-platform`

原因：

- 英文表达稳定
- 不局限于仓库层或 ETL 层
- 后续外部系统接入时更自然

## 7. 数据中台分层设计

### 7.1 L0 原始层 Raw Layer

职责：

- 保存原始请求/响应
- 保存 API 调用审计
- 保存字段回放依据
- 保留所有返回字段，不丢字段

必须支持：

- 通过 endpoint + store + businessDate + traceId 回放原始返回
- 对齐 API 文档字段
- 字段获取覆盖度检查

这层的目标不是给业务读，而是：

- 防丢
- 审计
- 排错
- 字段治理

---

### 7.2 L1 标准事实层 Canonical Fact Layer

职责：

- 保存标准化后的业务事实
- 覆盖全部门店、全历史、全业务日
- 成为中台的唯一业务事实真相源

内容包括：

- 门店 / 组织
- 会员
- 卡
- 消费主单 / 支付 / 明细
- 充值主单 / 支付 / 赠券 / 销售归属
- 账户流水
- 员工
- 员工项目能力
- 上钟明细 / 汇总
- 提成明细
- 提成配置 / 配置明细

原则：

- 所有需要被过滤、关联、聚合、审计、补数验证的字段，必须进入事实层结构化字段范围
- 长尾字段可继续保留 payload_json，但必须知道其归属对象

---

### 7.3 L2 状态与质量层 State & Quality Layer

职责：

- 表达同步状态
- 表达历史 run
- 表达完整性判定
- 表达字段覆盖度
- 表达补数任务状态

这层必须清楚区分：

1. 历史执行事实
   - ingestion_run
   - ingestion_endpoint_run
2. 最新可用状态
   - sync_job_state（重定义后）
3. 可答性状态
   - capability_completeness_snapshot
4. 字段覆盖与质量状态
   - 字段获取覆盖审计
   - schema alignment 审计

原则：

- 历史过程和当前状态绝不能混用
- partial / failed / pending 必须有明确原因

---

### 7.4 L3 主题服务层 Theme & Projection Layer

职责：

- 为未来上层提供默认读取入口
- 固定稳定的业务语义
- 对外隐藏事实层复杂性

主题层至少包括：

- HQ 总览
- 门店总览
- 充值消耗
- 会员洞察
- 员工看板
- 财务摘要
- 异常解释
- 目录与详情对象

原则：

- 上层默认只读这层
- 只有少量高级审计/对账/新主题开发可以直读事实层

---

## 8. 读取策略

推荐策略：

> 事实层是真相源，主题层是默认读取入口，原始层是回放与审计层，状态层是可用性解释层。

### 默认读取规则

未来上层默认读取：

- 主题层 / projection / service objects

### 特例读取规则

以下场景允许读取事实层：

- 数据审计
- 对账
- 补数验证
- 主题开发
- 特殊高级分析

### 禁止模式

未来上层不应该：

- 每个业务问题都直接拼装事实表
- 自己推断最新业务日
- 自己推断能否回答
- 自己绕过 completeness / projection

---

## 9. API 文档全字段纳管方案

### 9.1 原则

API doc 字段必须全部纳管。

这里的 API doc 不仅指外部网页，也指已经被整合到 Navly 仓内的文档副本，尤其是：

- `docs/api/qinqin/auth-and-signing.md`
- `docs/api/qinqin/endpoint-manifest.md`
- `docs/api/qinqin/member/*.md`
- `docs/api/qinqin/staff/*.md`

未来字段台账、事实层模型、主题层字段选择，都必须从这些仓内文档反推，而不是凭实现代码猜测。

这不意味着所有字段都立即结构化，但意味着每个字段都必须知道：

- 来自哪个 endpoint
- 是哪个 field path
- 当前是否已获取
- 当前在哪一层落地
- 当前是否已结构化
- 当前是否被主题层使用
- 如果未使用，为什么未使用

### 9.2 字段治理三层

#### Tier A：必须结构化字段

进入事实层结构化列的字段。

判定条件：

- 会用于过滤
- 会用于 join
- 会用于聚合
- 会用于 completeness
- 会用于 projection
- 会用于对账和补数验证

#### Tier B：必须留存字段

不一定要结构化，但必须保留在原始层 / payload 中。

判定条件：

- 未来可能用于建模
- 排错需要
- 审计需要
- 文档必须对齐

#### Tier C：暂不使用字段

也必须登记，但可暂不纳入事实或主题。

要求：

- 需要有明确标记
- 不能默默消失

### 9.3 当前现实

当前工作簿已经表明：

- 响应字段总数：459
- not_landed：0
- 说明字段没有彻底丢失

但同时：

- `structured_promoted` 只有 87
- 大量字段仍只是 `payload_json_preserved`

因此，数据中台第一阶段必须把“字段都接住了”升级成“字段都需求化了”。

## 10. 全部门店 + 全历史回溯策略

### 10.1 目标

- 覆盖全部目标门店
- 覆盖全历史业务日
- 能稳定重跑与校验

### 10.2 当前问题

当前已经观察到：

- `sync_job_state` 与 `ingestion_endpoint_run` 存在错位
- 历史 backfill 倒序写导致最新状态被旧业务日覆盖
- 某些 endpoint 状态语义不清，例如 `commission_setting_list = partial`

### 10.3 中台要求

全历史回溯必须区分：

1. 历史执行记录
2. 当前最新可用状态
3. 回溯 cursor 进度
4. 当前可答状态

不能再用一张表同时表达以上 4 件事。

### 10.4 建议策略

- 保留 `ingestion_run / ingestion_endpoint_run` 作为历史执行事实
- 重定义或重建 `sync_job_state` 作为“最新可用状态表”
- 回溯执行进度单独管理
- 回溯结束后通过 reconcile 更新最新状态，而不是让历史 cursor 直接覆盖最新状态

---

## 11. 当前已确认的数据层问题

### 11.1 最新状态表语义错误

问题：

- `sync_job_state` 当前更像“最后处理到哪个业务日”
- 而不是“最新可用业务日状态”

现象：

- 历史 run 到了更新日期
- sync state 却停在更老日期

### 11.2 `commission_setting_list` 状态不可信

问题：

- 配置缺失时返回空数组 + warning
- warning 没绑定 endpointId
- ingestion 汇总后落成 `partial`
- 但本质上不清楚是 auth 缺失、业务空返回、还是真实 partial

### 11.3 `store_member_insight` 依赖并未真正判定

问题：

- 当前 `table:*` dependency resolver 未实现
- completeness 将其标记为 pending
- 实际不是“等待数据”，而是“等待代码实现”

### 11.4 projection 只是 completeness 的被动翻译

问题：

- projection 当前缺少更细粒度 reason
- 不足以作为未来统一消费入口

### 11.5 事实表存在历史双轨

线上已观察到：

- `warehouse_record`
- `warehouse_records`

新中台必须消除事实表真相源歧义。

## 12. 代码迁移策略

### 12.1 不搬旧目录整体，只搬稳定模块

迁移对象优先级：

#### 可优先复用

- `connector-qinqin-real`
- `data-ingestion`
- `data-warehouse`
- `completeness-plane`
- `answer-plane`
- 与 Qinqin API 字段/endpoint 相关 contracts

#### 复用前必须重构

- `sync_job_state` 相关逻辑
- backfill 相关逻辑
- completeness resolver
- projection 状态解释
- 字段映射与字段治理

#### 暂不纳入数据中台核心

- 旧 runtime 业务编排层
- skill / prompt / query router
- 深度分析插件
- 上层 tool UX

### 12.2 新目录中的模块边界

建议分成以下包：

- `contracts/`：数据中台 contracts
- `ingestion/`：API 拉取、窗口、回溯、调度
- `warehouse/`：事实层落库与查询
- `quality/`：字段审计、数据质量、状态 reconcile
- `completeness/`：能力可答判断
- `projections/`：主题构建
- `serving/`：对外消费对象
- `migration/`：旧表/旧逻辑清理与迁移脚本

## 13. 分阶段实施方案

### 阶段 1：新目录建骨架

目标：

- 建立 `platforms/data-platform`
- 放入 README、模块说明、迁移规则
- 明确新旧代码边界

产出：

- 新目录结构
- 模块职责说明
- 迁移清单

---

### 阶段 2：迁入事实层与同步层

目标：

- 把 Qinqin endpoint contract、ingestion、warehouse 迁入新目录
- 修正 sync state 语义
- 修正历史回溯写状态逻辑

产出：

- 新的事实层可跑
- 新的状态层可信

---

### 阶段 3：迁入 completeness 与 projection

目标：

- 补 table dependency resolver
- 修 `commission_setting_list` 状态语义
- 让 projection 可解释

产出：

- 数据中台可答状态闭合

---

### 阶段 4：字段全量需求化

目标：

- 以 API doc 为边界做字段三级分类
- 产出字段治理台账
- 明确哪些字段继续提升为事实层结构化字段

产出：

- 字段治理文档
- 结构化字段路线图

---

### 阶段 5：全门店全历史治理

目标：

- 全部门店回溯
- 全历史状态统一
- 建立每日覆盖与质量巡检报表

产出：

- 真正的数据中台运行基线

---

## 14. 验收标准

### 14.1 结构验收

- 新数据中台代码位于独立目录
- 目录与旧上层解耦
- 模块职责清晰

### 14.2 数据验收

- 全白名单 / 目标门店完成纳入
- 全历史回溯可执行
- `sync_job_state` 与历史 run 不再冲突
- `commission_setting_list` 状态语义清晰
- `store_member_insight` 不再因 resolver 未实现卡住

### 14.3 字段验收

- API doc 全字段有台账
- 每个字段有落点
- 不再存在“字段接住了但没人知道怎么用”的灰区

### 14.4 消费验收

- 上层默认可以只读主题层
- 主题层结果都能追溯到事实层与状态层

---

## 15. 非目标

当前方案明确不做以下事情：

- 不在本阶段重构旧业务上层
- 不在本阶段重构 skill / tool / prompt 体系
- 不在本阶段重写 OpenClaw 权限内核本身，只对其在 Navly 中的保留边界和接入方式做收口
- 不在本阶段追求所有上层体验 bug 清零

这些都应在数据中台与权限内核稳定后进行。

## 16. 推荐下一步

建议紧接着产出一份实施计划，至少包含：

1. 新目录落地方案
2. 旧模块迁移优先级
3. 状态层修复顺序
4. 字段治理分工
5. 全门店全历史回溯执行计划
6. 上线切换策略

---

## 17. 核心判断

这个项目接下来不应该被表述为：

- “修数据模块”
- “继续补旧系统数据层”

而应该被表述为：

> 建设一个新的、可长期保留的数据中台；旧数据代码只是原材料，只有经过重构并纳入新分层后，才算进入长期资产范围。
